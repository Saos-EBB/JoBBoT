## 2026-07-17 — chore(config): disable ams source, blocked by anti-bot token + robots.txt

**Was:** `npm run scrape:ams` scheiterte (auch standalone, nicht UI-spezifisch) mit `Timeout ... waiting for event "response"` in allen drei Queries. Debug-Playwright-Run zeigt: AMS liefert für `/api/statusinfo`, `/api/frontend/aem-content`, `/api/frontend/predefined-searches` und `/api/search` jetzt `401 Unauthorized` — jeder Request braucht einen client-seitig signierten `x-ams-access-token`-Header (Algorithmus steckt im minifizierten `main.js`-Bundle), den unser Scraper nicht mitliefert. Zusätzlich verbietet `robots.txt` explizit `Disallow: /public/emps/` für alle User-Agents außer `LinkedInBot` — das ist keine zufällige Frontend-Änderung, sondern eine bewusste Sperre gegen automatisierten Zugriff. Token-Mechanismus wird deshalb nicht nachgebaut (würde robots.txt + Anti-Bot-Schutz gleichzeitig umgehen). `ams` in `config/sources.json` auf `enabled: false`, `scrapers/ams.ts` bleibt unverändert liegen für den Fall, dass AMS den Zugriff wieder öffnet.
**Nicht gebaut:** kein Ersatz-Scraper für AMS-exklusive Inserate — laut AMS speist sich `jobs.ams.at` aus eJob-Room-Selbstbedienung, direkt gemeldeten Firmenanforderungen, aus dem Web aggregierten Firmen-/Behörden-Inseraten und teils sogar Inseraten der deutschen Bundesagentur für Arbeit. Kein 1:1-Ersatzkanal, also nichts Neues gebaut, nur recherchiert.
